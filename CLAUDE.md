# yulik3d

## Описание проекта

**yulik3d** — интернет-магазин фигурок и макетов, напечатанных на 3D-принтере. Сайт делается для малого бизнеса (один продавец, он же админ), выросшего из хобби 3D-печати.

### Предметная область

Продавец изготавливает на заказ:
- **Фигурки** (category type: `figure`) — персонажи из игр, фильмов, мультфильмов, аниме. Ручная работа + 3D-печать, полимерная смола, производство 20–50 дней.
- **Макеты** (category type: `other`) — декор, аксессуары, кастомизация ПК, вазы, горшки и т.п.

Каждый товар может предлагаться в разных **размерах** и с доп. **опциями** (покраска, гравировка и др.) — каждая опция имеет собственную доплату к базовой цене.

### Ключевой пользовательский функционал

- Каталог с фильтрацией по категориям/подкатегориям, поиск.
- Карточка товара: галерея, markdown-описание («Информация о товаре» + «Особенности»), выбор размера и опций с живым пересчётом цены.
- Корзина (frontend-only, localStorage) — доступна только авторизованным.
- Регистрация / авторизация.
- Избранное — только для авторизованных.
- Оформление заказа (только авторизованным) → заказ сохраняется в БД → админ видит его в админке → админ связывается с покупателем сам (оплаты в MVP нет).
- История заказов пользователя со статусами.

### Админский функционал

- CRUD товаров, картинок, опций, категорий/подкатегорий (с выпадающими списками из БД и возможностью создать новое по ходу).
- Скрытие товара (`item.hidden`): товар пропадает из каталога, но остаётся доступен по прямой ссылке с пометкой «Не доступно к заказу».
- Просмотр заказов, ручной перевод статуса вперёд: `Создан → Подтверждён → На изготовлении → В доставке → Завершён`. Отдельно — `Отменён`.
- Роль `admin` выдаётся **вручную через БД** (UI для назначения роли не делаем).

### Стек

- **Фронт:** SPA. Черновая реализация на моках (TS + Vite + Handlebars) лежит в соседней папке `yulik3d_frontend/` — **вне** этого проекта, используется только как референс. Финальный стек и структуру определим на этапе фронта.
- **Бэк:** Go, чистая архитектура, без веб-фреймворков. Сырой SQL (не ORM).
- **БД:** PostgreSQL.
- **Хранилище картинок:** MinIO (в БД — object key, публичный бакет для MVP).
- **Почта:** SMTP-уведомление админу при новом заказе. Реализуется после ядра, в MVP не обязательна для запуска.
- **Контейнеризация:** Docker Compose в корне проекта, отдельные `Dockerfile` для бэка и фронта.

### Платёжная логика

В MVP оплата **не реализуется**. Заказ создаётся в статусе `Создан`, админ связывается с покупателем в соцсетях и двигает статус вручную.
